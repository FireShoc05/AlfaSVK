# Документация проекта AlfaSVK для подбора хостинга

Этот документ содержит техническое описание проекта для подбора оптимального российского хостинга.

## 1. Общая информация о проекте
- **Тип архитектуры:** Single Page Application (SPA)
- **Сфера применения:** Web-приложение с дашбордами/отчетами (присутствует экспорт в Excel).
- **Текущее место хостинга:** Vercel (о чем свидетельствует файл `vercel.json` с настройками редиректов).

## 2. Технологический стек фронтенда
Проект представляет собой классическое React-приложение, собираемое с помощью Vite.
- **Фреймворк:** React 19 (`react`, `react-dom`)
- **Сборщик (Bundler):** Vite 8
- **Маршрутизация (Routing):** React Router DOM v7
- **Управление состоянием:** Zustand
- **Дополнительные библиотеки:** `xlsx` (для генерации отчетов в Excel), `lucide-react` (иконки).
- **Стилизация:** Ванильный CSS.

## 3. Бэкенд и База Данных
Проект **не имеет** собственного серверного бэкенда (Node.js/Express и т.д.). В качестве Backend-as-a-Service (BaaS) используется **Supabase**.
- **Интеграция:** `@supabase/supabase-js`
- Вся логика работы с данными, аутентификацией и базой данных (PostgreSQL) делегирована Supabase.
- Для работы приложения требуются переменные окружения (например, `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`).

## 4. Требования к российскому хостингу

Для корректной работы приложения новому хостинг-провайдеру необходимо удовлетворять следующим требованиям:

### 4.1. Требования для хостинга Фронтенда (Обязательно)
Поскольку это SPA на Vite, сам сайт — это набор статических файлов (HTML, CSS, JS). 
Подойдет любой хостинг статики или VPS со следующими возможностями:
1. **Поддержка SPA-маршрутизации (Catch-all fallback):** Сервер должен уметь перенаправлять все запросы (кроме существующих файлов) на `index.html`. На Vercel это делалось через `vercel.json` (`"source": "/(.*)", "destination": "/index.html"`). В Nginx это правило `try_files $uri $uri/ /index.html;`.
2. **Среда сборки (опционально, но желательно):** Наличие Node.js (v18+) для автоматической сборки проекта через `npm install` и `npm run build` прямо на сервере. В противном случае придется собирать проект локально и загружать папку `dist` по FTP/SSH.
3. **Поддержка HTTPS (SSL-сертификаты):** Обязательно (желательно бесплатные от Let's Encrypt), так как приложение работает с внешним API (Supabase).
4. **Настройка переменных окружения (ENV):** Возможность безопасно задать ключи доступа к API при сборке.

### 4.2. Вопрос локализации базы данных (Supabase)
*Важно прояснить этот момент при подборе:*
В данный момент используется Supabase. Если цель переезда — **только** перенос фронтенда на российские серверы (например, для ускорения загрузки статики или обхода блокировок Vercel), то подойдет любой провайдер (Timeweb, Рег.ру, Selectel, Яндекс Cloud).
Если же цель переезда — **полное импортозамещение и соблюдение 152-ФЗ** (хранение персональных данных пользователей в РФ), то потребуется отказаться от облачного Supabase и развернуть собственную архитектуру на российском VPS/VDS (например, self-hosted Supabase, Appwrite или связка PostgreSQL + кастомный бэкенд).
