# СИСТЕМА АВТОРИЗАЦИИ И ПАНЕЛЬ АДМИНИСТРАТОРА

Этот план описывает процесс внедрения полноценной закрытой экосистемы для приложения: системы ролей (Администратор, Обучающийся, Агент), базы данных сотрудников и промежуточных экранов (онбординг).

> [!IMPORTANT]
> База данных сотрудников временно (до подключения реального бекенда) будет храниться в `localStorage` (браузере устройства, на котором создавались пользователи). Чтобы зайти под одним из созданных сотрудников с другого устройства, админу пока придется создавать их на том же устройстве или нам нужно будет прикрутить облачную БД (например Firebase/Supabase). Однако для текущего прототипа и тестирования логики локального хранилища хватит с головой.

## Проектируемая архитектура состояний
1. **`useUsersStore` (НОВОЕ ХРАНИЛИЩЕ)**: Будет эмулировать базу данных сотрудников. Способно добавлять пользователей, хранить их хэши логинов, статусы и контактные данные.
2. **`useAuthStore` (ОБНОВЛЕНИЕ)**: По умолчанию `isAuthenticated` станет `false`. Будет управлять сессией активного пользователя.

## 1. Роутинг и потоки (App.jsx)
Мы полностью изменим систему маршрутизации приложения.

### [NEW] Структура путей:
- `/login` — Публичный экран входа (Логин + Пароль).
- `/onboarding` — Окно доп. информации при первом входе сотрудника (Телефон / ТГ / Почта).
- `/training` — Экран-заглушка "Пройди обучение" для стажеров.
- `/admin` — Панель админа.
- Основной интерфейс (Meeting, Profile) будет обернут в защищенный слой `<ProtectedRoute>`.

## Verification Plan
1. Вручную зайти под встроенным админом.
2. Создать сотрудника со статусом обучающегося и скопировать его пароль.
3. Разлогиниться, войти под стажером -> проверить онбординг -> проверить заглушку обучения.
4. Зайти под админом, сбросить пароль стажера, сменить статус на "Действующий".
5. Зайти под стажером с новым паролем -> проверить доступ в рабочее пространство.
